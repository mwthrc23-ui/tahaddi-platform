# نموذج الصلاحيات — تحدّي

> النموذج: RBAC للأدوار العامة وABAC للملكية والمؤسسة وحالة المورد والخطة.

الوثائق المرتبطة: `architecture.md` لحدود الخدمات والعقود، و`user-flows.md` لمسارات التنفيذ.

## 1. مبادئ

1. المنع هو الافتراضي.
2. التحقق على الخادم لكل HTTP وWebSocket command.
3. الدور لا يكفي وحده؛ يجب التحقق من الملكية و`organizationId` وحالة المورد وحدود الخطة.
4. لا تمنح عضوية مؤسسة وصولًا إلى مؤسسة أخرى يشارك فيها المستخدم.
5. حساب مدير النظام لا يستخدم للمهام اليومية، وتُسجل إجراءاته الحساسة.
6. لا يمكن للمستخدم ترقية دوره أو تغيير مالك المورد عبر payload عادي.
7. القراءة العامة تستخدم DTO منقحًا، لا كيان قاعدة البيانات كاملًا.

## 2. الأدوار

| الدور | النطاق | الوصف |
|---|---|---|
| Visitor | عام | مستخدم غير موثق |
| GuestParticipant | جلسة واحدة | ضيف يحمل صلاحية مشاركة قصيرة العمر |
| Player | حساب | مستخدم مسجل وقدراته الشخصية |
| Host | مورد/حساب | مالك مسابقة أو مقدم مخول لجلسة |
| OrgMember | مؤسسة | عضو عادي في مساحة مؤسسة |
| OrgEditor | مؤسسة | ينشئ ويدير محتوى المؤسسة |
| OrgAdmin | مؤسسة | يدير الأعضاء والمحتوى والتقارير |
| OrgOwner | مؤسسة | كامل إدارة المؤسسة والفوترة ونقل الملكية |
| Moderator | المنصة | مراجعة البلاغات والمحتوى |
| Support | المنصة | قراءة محدودة للدعم دون أسرار أو تعديل نتائج |
| SystemAdmin | المنصة | إدارة النظام الحساسة مع تدقيق قوي |

`Host` قدرة مشتقة من ملكية المسابقة/الجلسة أو تفويض صريح، وليست شارة تمنح السيطرة على كل المسابقات.

## 3. سمات التفويض ABAC

- هوية الفاعل وحالته وMFA للعمليات الحساسة.
- `organizationId` النشط وعضويته وحالتها.
- مالك المورد أو منشئه أو تفويضه.
- حالة المورد: مسودة، منشور، نشط، مؤرشف.
- خطة الاشتراك وحدود الاستخدام.
- علاقة المشارك بالجلسة وحالة الحظر.
- غرض token وانتهاؤه و`sessionId` المربوط به.

## 5. سياسات الموارد

### Question وQuiz

- المورد الشخصي: `ownerUserId === actor.userId`.
- مورد المؤسسة: `resource.organizationId === activeOrganizationId` وعضوية فعالة بصلاحية مناسبة.
- بعد بدء جلسة لا يعدل المصدر المؤثر على الجلسة؛ تقرأ الجلسة Snapshot.
- الأرشفة لا تكسر التقارير أو الجلسات السابقة.

### QuizSession

- إنشاء الجلسة يتطلب مسابقة منشورة وملكية/تفويضًا وحدًا متاحًا بالخطة.
- أوامر التحكم تقبل من Lease holder للمقدم الرئيسي مع `expectedVersion`.
- شاشة العرض تحصل على token قراءة محدود لا يسمح بأوامر الإدارة.

### ParticipantAnswer

- المشارك يكتب إجابته فقط للجلسة والسؤال الحاليين.
- لا يقرأ إجابات الآخرين الفردية أثناء اللعب.
- لا يعدل أو يحذف الإجابة بعد قبولها.
- مقدم المسابقة يرى إحصاءات مجمعة أثناء السؤال، والتفصيل بعد الجلسة حسب سياسة الخصوصية.

### Report

- اللاعب يرى ملخصه فقط.
- المقدم يرى تقرير جلسة يملكها.
- المؤسسة ترى تقارير مواردها.
- الدعم لا يرى إجابات أو بيانات شخصية إلا بتذكرة وحالة وصول موقوتة ومسجلة.

## 6. رموز الجلسة والضيف

### رمز الغرفة

- معرف اكتشاف قصير، لا يمثل سر مصادقة ولا يستخدم كمفتاح قاعدة بيانات.
- Rate limit لمحاولات البحث، ورسالة موحدة للرمز الخطأ والمنتهي.

### Guest Token

Claims الدنيا: `sub=participantId`, `sessionId`, `purpose=play`, `iat`, `exp`, `jti`.  
لا يحتوي الاسم أو النقاط أو الدور الإداري. يجدد عبر Resume Token دوّار ويُلغى عند الحظر أو نهاية فترة الاستئناف.

### Display Token

قراءة فقط، مرتبط بجلسة واحدة، يعرض payload عامًا منقحًا ولا يستطيع إرسال أوامر مقدم.

## 7. حراس الطلب والأحداث

ترتيب التحقق الإلزامي:

1. صحة البروتوكول والحجم وSchema.
2. المصادقة وصلاحية token والغرض.
3. حالة المستخدم/العضوية/المشارك.
4. تطابق المؤسسة وملكية المورد.
5. الصلاحية الدقيقة وحالة المورد.
6. حد الخطة وRate Limit.
7. idempotency ونسخة الحالة.
8. تنفيذ المعاملة ثم Audit/Outbox.

فشل أي خطوة يعيد code ثابتًا ورسالة عربية آمنة، دون stack trace أو كشف وجود مورد خاص.

## 8. العمليات الحساسة

تحتاج إعادة مصادقة أو MFA عند تفعيله:

- نقل ملكية مؤسسة.
- تغيير أدوار OrgAdmin/Owner.
- حذف حساب أو تصدير بياناته.
- تغيير إعدادات أمن النظام.
- دخول طارئ لمدير النظام أو دعم إلى بيانات مستخدم.
- اعتماد جائزة أو عملية مالية مستقبلًا.

## 9. سجل التدقيق

يسجل على الأقل:

- actor، الدور الفعلي، organizationId، requestId، IP مختصر/مجزأ وفق السياسة.
- action، resourceType/resourceId، النتيجة، reasonCode.
- before/after منقحين من الأسرار والبيانات غير اللازمة.
- الوقت من الخادم وسياق التفويض أو التذكرة.

أحداث إلزامية: تغيير الأدوار، النشر، بدء/إنهاء الجلسة، استبدال المقدم، الطرد والحظر، تعليق الحساب، قرارات الإشراف، تغيير النتائج الإداري، وإعدادات النظام.

## 10. اختبارات الصلاحيات المطلوبة

- جدول اختبارات allow/deny لكل صف حرج في المصفوفة.
- IDOR عبر تغيير معرف المورد.
- مستخدم عضو في مؤسستين مع تبديل `organizationId`.
- عضو أزيلت عضويته واتصاله WebSocket ما زال قائمًا.
- Guest Token لجلسة يستخدم في جلسة أخرى.
- Display Token يحاول إرسال أمر مقدم.
- Host سابق يحاول التحكم بعد انتقال Lease.
- Moderator يحاول تعديل نتيجة أو AuditLog.
- حساب معلق عبر REST وWebSocket واتصال مستعاد.
- تجاوز حد الخطة بطلبين متزامنين.
